## 1. 原著の記述形式に対する翻訳の書き方ルール

原著の記述形式に対して翻訳の書き方をまとめます。主に繰り返し発生する部分に対してまとめます。

下記のルールに従った各章のサンプルを[こちら](../doc_sample)に用意してますので参考としてください。

### 1.1 除外箇所

原著における以下の項目については、翻訳対象から除外します。

- 章最初：Key Concepts（各章のキーワードがかかれてます）
- 章最初：QuickLook（各章の概要紹介です）
- コラム：SafeHome（各章に0-4個くらいコラム形式で割込み記載されてます）

### 1.2 基本ルール

原著で繰り返し発生する表現に対する基本ルールを以下に記載します。

#### A) 章表現と文中の引用

markdown形式の階層で表現ください。

文書内では第X章、X.X節、X.X.X項と呼びます。Re:VIEW版で実施しようとした記法での対応（＋電子書籍向けリンク作成）は行いません。

```
# 第X章　ソフトウェアXXX

## X.1 ソフトウェアXX

### X.1.1 ソフトウェアXX
```

#### B) 句読点、箇条書き、先頭太字（調整中）、

句読点としては「、」「。」を使用します。なお、翻訳の参考となる第6版の句読点は「，」「．」です。textlintでチェックが行われますが、コピペで参照する際には注意してください。第6版情報を使用した場合、一括変換なりが必要となります。

**B-1 数字有箇条書き**

数字箇条書きについては、ひとまずMarkdownの記法に従う形式とします。

1. リスト1
2. リスト2

**B-2 数字無し箇条書き（調整中）**

単な箇条書きについては、レビューの観点からひとまずMarkdownの記法に従う形式とします。最終成果物として提出時の修正なりは調整中。

- 箇条書き1
- 箇条書き2

**B-3 先頭太字（調整中）**

先頭太字に関してはMarkdown形式とは外れますが、以下とする感じです。（調整中）

スペースで空ける表現となります。lintチェックで怒られる部分など調整が必要。

----
lbl 太字対象 太字以降の文字

----

#### C) 原著注釈と訳注（調整中）

**C-1 まとめ方針（調整中）**

- 原著注釈と訳注は、章単位でインクリメントさせ、[1]といった番号で示してください。
- 注：原著注釈と訳注を混ぜた番号をインクリメントするので、訳注が入ると原著との番号が変わります
- 原著注釈と訳注はがある段落の直下に、「 [1] XXX 」といった記述を行います。
- 訳注の場合には、説明側で「 [2] [訳注1] XXX 」などと記述ください。

**C-2 原著注釈について**

> （元記述）原著の注釈は、[1]で表現します。また、直後の段落に翻訳を入れておいてください。

> 原著の注釈を、{1}で表現することによって、InDesignで変換させる。半角全角調整中。あわせて、Noのインクリメント調整中。今のところ、章単位で1から番号をつけてください。

**C-3 訳注について**

原著訳注と同等の扱いです。説明側で「[訳注1]」などとつけてください。

以下が例となります。訳注が入るので原著の[1]が[2]に変わります。

-----
このようなアジャイル開発[1]の基礎をなす考え方は、アジャイル手法[2]へと繋がった。アジャイル手法は従来のソフトウェアエンジニアリングで実際に直面する弱点を克服するために開発された。アジャイル開発は、大きな利益をもたらす可能性がある。しかし、あらゆるプロジェクト、プロダクト、人々、状況において利用できるとは限らない。また、アジャイルは厳格なソフトウェアエンジニアリングのプラクティスを否定するものでもなければ、ソフトウェアに関する作業の概念を覆すものでもない。

[1] [訳注1] アジャイル（Agile）という単語は「俊敏な、敏捷な」という意味だが、ソフトウェアエンジニアリングの文脈では手法や体系の意味もある。アジャイル開発は俊敏な開発を指すが、アジャイル体系における開発手法という意味にもなる。アジャイルという単語は多数登場するので慣れよう。

[2] アジャイル手法は、軽量な、もしくは無駄のない（lean）方法論と呼ばれることがある。

-----

#### D) 人名

人名は原著記載の英語名で記載ください。

#### E) 参考文献

参考文献についても、原著の表現で”[  ]”で囲んで表現します。
巻末に別途原著とあわせてまとめます。

（論点が異なりますが、翻訳された日本語版書籍の紹介については調整中です…）

D-Eについては、以下が例となります。

-----
最新のコンピュータシステムが構築される際に、プログラマひとりの時代と同じような疑問が相変わらず問われている[2]。

[2] ソフトウェアビジネスに対する良書[DeM95]にて、Tom DeMarcoが次のように反論している。「なぜソフトウェアのコストがそんなに高いのか」と疑問を投げるのではなく、「コストを低く抑えるためには何をすべきだったか」という疑問を持ち始めるべきだ。

#### F) 引用個所

引用個所は出版社として明示しておくべき部分となりますので、できる限り明確にわかるよう引用表現とします。

**F-1 文単位の引用**

原著には書籍からの引用が多く段落単位で記載されています。原著には明確に引用とわかる段落（フォントが異なる）がありますので、その箇所にはmarkdown「>」形式の引用記述をお願いします。

以下が例です。

-----
*IEEE Software*のジャーナル特別編において、ゲスト執筆者らが次のような洞察[deS09]を示している。

>ソフトウェアエンジニリングは、ソフトウェア開発プロセスとプロダクトの双方を向上させるために作られ、関連する技術、ツール、手法が多数存在する。次々と新しい技術が登場し、使うに値する結果をもたらしている。

-----

**F-2 文中斜字表現での明らかな引用**

文中に明らかとわかる斜字での引用が含まれる場合があります。この場合、インラインコード「`」で囲んでください。ただ、こちらは正直わからない部分が多いので、無理してつける必要はありません。

以下が例です。

-----
真実を認めることは困難なことが多い。問題を指し示す際には、多くの人はこの傲慢さに苦しんでいる。数秒聞いて、そして考える。`理解したよ。それではこの問題を解決しようか`。不幸なことに、大抵の問題は理解が簡単ではない。次の質問の回答に少し時間を費やす価値はあるだろう。

- `誰がその問題解決について利害があるか`。すなわち、誰がステークホルダか。
- `何が分かっていないのか`。問題を正しく解決するためにはどんなデータや機能、フィーチャが必要であるか。

-----


#### G) 斜字・修飾表現（★調整中…皆さんが訳を開始した後で再議論★）

原著には特に斜字表現が多数あります。以下のようなパターンがあるようです。

**G-1 参考：原著の斜字や太字などの修飾表現**

- 「Peopleware（ピープルウェア）」「How to solve it」など文中に特定の書籍名を記載する場合
- 明らかに書籍などを引用している箇所
- （Modeling章）モデル図上などの（プログラム）関数名を文中で表現
- 索引、章初めのKeyConceptと対応させているもの
- 特定の単語（初出など？）を際立てたいもの
- 特定（単語、文）の強調

特に後半部分ついては原著側で乱用気味にも見えますので、必須と担当者任せに分けて、以下にしたいと考えてます。（特に後者はルールで縛りは入れません、各翻訳担当・部の範囲で統一くらいとします）

**G-2 斜字や太字などの修飾表現：必須対応ケース**

斜字や太字などの修飾表現について、必須となる対応ケースは以下の3点とします。

1. （上記F 引用個所 参照）文中に明らかに書籍を引用している箇所：インラインコード（'1つで囲む）表現とする
2. 特定の書籍名：英語名のままイタリック（*1つで囲む）表現とし、括弧は不要
3. （プログラム）関数名を文中で表現：イタリック（*1つで囲む）表現とする

以下が例です。

-----
近代的なコンピュータが存在する前に書かれた古典的な著書である*How to solve it*[Pol45]のなかでGeoge Polyaは問題解決の本質について述べている。これはソフトウェアエンジニアリングプラクティスの本質ともいえる。

1. `問題を理解する`（コミュニケーションと分析）
2. `解決策を計画する`（モデリングとソフトウェア設計）

-----

**G-3 上記以外の斜字や太字などの修飾表現への対処（調整中）**

- すべて対応する必要はありません
- 日本語に斜字（イタリック）は読みづらいので使用しないようにします
- 「」もしくは太字で対処することとします
- 索引対象は下記「I) 索引向けの単語」を参考にHTMLタグを入れます、太字等は任意です
- メモ：6版日本語のパターンを後で調べる

### 1.3 図表、コラム

図表やコラム等の記述方法をまとめておきます。

#### A) 図

図に関しては、以下どちらかでお願いします。

1. 直接EPSファイルを修正して翻訳された図データにする
2. 文書内でコードブロック形式で英日の対訳を入れる

2の形式に関しては以下が例となります。
```
図1.1 ハードウェアの故障曲線
Failure rate 故障率
Time 時間
"Infant motality" 初期の故障
"Wear out" 劣化
```

#### B) 文書内の補足が記述された箱（コラム？サイドバー？呼び方がわかんない…）

文書内の補足が記述された箱に関しては、Markdown表現とは異なりますが、以下のように「^^^^」で囲んで表現ください。

囲み部分内の装飾はMarkdownで可能です。

^^^^

タスクセット

タスクセットは、ソフトウェアエンジニアリングのアクションの目的達成に向けた実作業と定義する。
たとえば、誘発（一般的には「要件収集（requirements gathering）」と呼ばれる）はコミュニケーションのアクティビティ内で発生する重要なソフトウェアエンジニアリングアクションである。
開発するソフトウェアに対してのさまざまなステークホルダの期待を理解することが要件収集のゴールである。

1. XXX
2. XXX

^^^^

#### C) 表（調整中）

表については、単にタイトルだけ入れて、同一のディレクトリにExcelで翻訳した表を入れておく方針を考えております。

サンプルは[こちら](../doc_sample/chXX.xlsx)

mdの文中には、コードブロック形式で英日の対訳を入れる形式としておきましょう。

以下がmd側の記述例となります。
```
表2.1 プロセスモデルの⽐較

ch02.xlsx参照
```

装飾は翻訳時は好きにして頂き、組版側に統一をおまかせしてよいと思います。
